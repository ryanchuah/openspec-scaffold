## Decision: All project state in tracked, agent-neutral files — no harness-native memory
**Date:** 2026-06-13
**Decision:** Project state lives only in tracked files (`AGENTS.md`, `ai-docs/`, `openspec/`). Agents must not rely on harness-native memory (Claude memory, OpenCode session memory, etc.) to carry project state across turns or sessions.
**Rationale:** This repo is worked by both Claude Code and OpenCode/DeepSeek agents; only tracked files are visible to both. A harness-native store would be invisible to the other harness and silently break cross-agent continuity.
**Alternatives considered:** Anthropic's native memory tool (reported ~39% agentic-search gain / ~84% token reduction in its Sept-2025 benchmark — figures per Anthropic, not independently verified) was rejected: it is Claude-only with no OpenCode equivalent, so it cannot be the shared state store for a dual-harness workflow. Revisit only if the dual-harness requirement is ever dropped.

## Decision: Workflow skills live only under `.claude/skills/` — not duplicated to `.opencode/`
**Date:** 2026-06-13
**Decision:** The OpenSpec workflow skills are kept as a single copy under `.claude/skills/openspec-*/SKILL.md`; they are NOT duplicated into an `.opencode/skills/` directory.
**Rationale:** OpenCode ≥ 1.16 auto-discovers `.claude/skills/**/SKILL.md` (gated by the env var `OPENCODE_DISABLE_CLAUDE_CODE_SKILLS`, default false), so both harnesses load the same skill files from one location. A second copy would create a divergence hazard (two sources drifting apart). Verified 2026-06-13 against opencode 1.17.4: the installed binary carries the `claude/skills/` scan path and the disable flag is unset.
**Alternatives considered:** A second `.opencode/skills/` copy, or a symlink `.opencode/skills/` → `.claude/skills/` — both rejected as redundant given the auto cross-load, and as a maintenance/divergence risk.

## Decision: Per-repo test command for commit-test-gate lives in scripts/test-cmd
**Date:** 2026-06-16
**Decision:** The commit-test-gate script (`scripts/test-gate.sh`, shared/byte-identical across repos) reads the per-repo test command from a one-line file `scripts/test-cmd`. When this file is absent, empty, or whitespace-only, the gate is a no-op (exit 0). When the command's executable cannot be resolved (e.g. fresh clone, missing `.venv`), the gate warns on stderr and allows the commit (exit 0) — it does NOT block on configuration errors that would be hostile to new clones. Only when the command runs and fails does the gate block (exit 2).
**Rationale:** A shared gate script avoids duplicating the test-gating logic across repos. The per-repo file is the single value the forthcoming single-source manifest will carry as a fill; it is intentionally NOT synced/shared. The config-error-as-no-op behavior prevents a missing interpreter from locking all commits in a fresh environment.
**Alternatives considered:** (a) An environment variable — rejected because it cannot be tracked/versioned per repo. (b) A symlink from each repo — rejected as fragile. (c) Blocking on config errors (failing closed) — rejected as hostile to new clones.

## Decision: Scaffold ships the commit-test-gate hook (carve-out from "No lifecycle hooks")
**Date:** 2026-06-16
**Decision:** The scaffold SHIPS a Claude Code `PreToolUse` hook (in `.claude/settings.json`, on `Bash(git commit*)`) that runs `scripts/test-gate.sh` to block commits whose tests fail. This is a deliberate, operator-approved exception to the "No lifecycle hooks in the scaffold" decision above.
**Rationale:** A deterministic tests-green-before-commit gate is high enough value (it closes the "executor claims tests pass" trust gap at the single chokepoint — the orchestrator's commit) to accept Claude-harness coupling for this one mechanism. The gate degrades safely: it is inert wherever `scripts/test-cmd` is absent (incl. the scaffold itself, which has no tests), so shipping it in the template costs nothing there while making it present for instantiated projects. Part of the `harden-delegation` change.
**Residual / known gap:** enforcement is **Claude-only** for now — OpenCode-driven commits are NOT gated until the deferred OpenCode plugin (v2). The harness-neutral piece (`scripts/test-gate.sh` + the `scripts/test-cmd` convention) is shared; only the wiring is Claude-specific.
**Post-hoc (2026-06-16, harden-instruction-surface):** AGENTS.md's cross-agent-compatibility section previously called `.claude/settings.json` "hook-free" — an instruction-audit defect, since the file ships a tracked `PreToolUse` hook. The stale claim was corrected; AGENTS.md now acknowledges the hook as a sanctioned, Claude-only carve-out running a tracked, agent-neutral script, cross-referencing this decision.
**Alternatives considered:** Re-scope so the scaffold ships only the neutral script and instantiated projects wire the hook (honoring the original decision) — considered and rejected by the operator in favor of shipping the hook in the template directly. A git `pre-commit` hook — rejected (`--no-verify` bypass; not a harness control).

## Decision: Executor/reviewer agents are NOT cross-loaded — both `.claude/agents/` and `.opencode/agents/` are required
**Date:** 2026-06-14
**Decision:** The delegated agents live in BOTH directories with different roles and formats — they are NOT consolidated to one copy the way skills are. `.opencode/agents/{apply-executor,archive-executor,openspec-reviewer}.md` (model `deepseek/...`, OpenCode `mode`/`permission` schema) are the PRIMARY executors, invoked via `opencode run --agent <name>` and used by both harnesses. `.claude/agents/{apply-executor,archive-executor}.md` (`model: sonnet`, Claude `tools` schema) are the FALLBACK, invoked via Claude's Task-tool `subagent_type` only when the opencode/deepseek run crashes. There is deliberately NO `.claude/agents/openspec-reviewer.md`: the reviewer must be a different model than the orchestrator, so it is deepseek-only (a Claude reviewer auditing Claude's own work defeats separate-model review).
**Rationale:** Unlike skills, agents do NOT cross-load. opencode 1.17.4 discovers agents from `.opencode/agents/` and `~/.config/opencode/agent` only — the binary carries no `claude/agents/` scan path and no agents opt-out flag (contrast `OPENCODE_DISABLE_CLAUDE_CODE_SKILLS`, which exists precisely because skills DO cross-load). The two formats are also not interchangeable (Claude `tools` + `model: sonnet` vs OpenCode `mode`/`permission` + `model: deepseek`), and the two roles select different models on purpose. So a single shared copy is impossible. Removing `.opencode/agents/` would break the deepseek primary path under BOTH harnesses (Claude's primary apply/archive/review shells out to `opencode run --agent`); removing `.claude/agents/` would remove the Sonnet resilience fallback for when opencode/deepseek itself is down. Verified 2026-06-14 against opencode 1.17.4 (binary scan-path inspection) and the agent files' frontmatter.
**Alternatives considered:** (a) Consolidate to one copy under `.claude/agents/` like skills — rejected: OpenCode cannot see `.claude/agents/`, and the formats/models differ. (b) Drop `.claude/agents/` and rely only on opencode — rejected: removes the fallback that exists for `opencode run` failures (the exact operational-crash case the apply/archive skills' failure ladders handle). See the companion "Workflow skills live only under `.claude/skills/`" decision above for the contrasting skills case.

## Decision: Change tiers are STANDING (in AGENTS.md); fast-track governs autonomy only
**Date:** 2026-06-14
**Decision:** The SMALL/MEDIUM/COMPLEX change-tier classification is a standing part of the normal workflow, documented in AGENTS.md `## Change tiers` (every change is classified; the operator initiates tier-2/tier-3 lifecycles). `ai-docs/fast-track-workflow.md` is slimmed to govern ONLY the autonomy override (proceeding without per-step checkpoints), which remains opt-in and trust-gated — dormant unless the operator explicitly grants it.
**Rationale:** Previously the tier definitions lived only inside the (opt-in) fast-track doc, so without a fast-track grant every non-trivial change ran the full lifecycle. Tiering (process-weight scaling) and autonomy (interaction cadence) are independent concerns: tiering should always apply (extrends's proven practice), while autonomy stays opt-in. Separating them lets agents right-size process by default while keeping the conservative phase-gated interaction default. This also harmonizes the golden source with extrends + psc-monitor so all three share these workflow sections.
**Alternatives considered:** Keeping tiers opt-in (bundled inside fast-track) — rejected: it forced full-lifecycle ceremony on small changes absent a grant, and diverged from extrends. Making autonomy standing too — rejected: autonomy stays opt-in per the operator's standing preference (Option A — no active grant in any repo).

## Decision: Apply-executor convergence detection is offloaded to a deterministic helper script
**Date:** 2026-06-16
**Decision:** The apply-executor's non-convergence detection (rules a and b) is computed by a deterministic Python helper script (`scripts/_convergence.py`) rather than relying on the executor model's in-context tracking. The helper normalizes error signatures (stripping line numbers, paths, timestamps, hex addresses, elapsed-time numbers), reads/updates durable per-change state (`/tmp/apply-convergence-<slug>.json`), and prints `CONTINUE` or `STOP:<a|b|c>:<detail>`. The executor pipes raw test output to the helper after each red test run and obeys the verdict. As of 2026-06-17 (W3 fix-convergence-guard, A3), the edited-file set for rule (b) is derived deterministically from `git diff` per-attempt SHA1 content-fingerprint deltas — `--editing` is now an optional fallback hint, no longer load-bearing for rule (b), because flash is unreliable at self-reporting `--editing` consistently across every iteration.
**Rationale:** Flash (the executor model) is unreliable at multi-rule textual normalization and at carrying exact state across 20+ tool calls — the original failure mode of the `p2-apply-c` looping incident. Offloading normalization to a deterministic script removes the precise-normalization burden from flash and makes rule (a)/(b) judgments reproducible rather than dependent on a specific model's in-context reasoning. The durable state file survives context-window resets, so the detection is robust even if the executor's context is truncated mid-run. Part of the `harden-delegation` change.
**Alternatives considered:** (a) Having flash normalize and track in-context — rejected; flash is unreliable at multi-rule textual normalization and at carrying exact state across many tool calls. (b) Asking flash to manually track file-edit counts — rejected; same in-context tracking problem.

## Decision: Structured NON-CONVERGENCE BLOCKER block as machine discriminator
**Date:** 2026-06-16
**Decision:** On STOP, the executor emits an exact 8-field `### NON-CONVERGENCE BLOCKER` block (fields: `trigger`, `task`, `test`, `signature`, `attempts`, `files`, `missing`, `suspected_cause`). The primary detects declared blockers by grepping the completion report for that literal heading; its absence means opaque give-up. In the apply-skill failure ladder (both Claude and OpenCode delegation paths), declared blockers are routed to orchestrator triage (tighten brief + fresh executor, escalate to user, or Sonnet by cause) instead of immediate Sonnet; opaque give-ups keep today's reflexive fallback behavior.
**Rationale:** Without a parseable discriminator, the primary would misroute every non-crash failure to reflexive Sonnet — defeating the purpose of stopping early on a genuine non-convergence that the plan/brief can fix. The heading is trivially grep-able, carries the diagnosis fields the orchestrator needs to triage, and applies to both delegation paths (Claude shelling out and OpenCode interactive). Part of the `harden-delegation` change.
**Alternatives considered:** Asking the primary to judge from a free-text completion report — rejected; ambiguous, non-grep-able, and the original failure mode (no discriminator → everything looks like a generic give-up).

## Decision: Reviewer budget raised to 780s with incremental output and partial salvage
**Date:** 2026-06-16
**Decision:** The reviewer timeout in `openspec-propose` (the only workflow that invokes the `openspec-reviewer`) is raised from 300s to 780s with soft-kill grace `-k 15`. The reviewer stays read-only (`edit: deny`, `bash: deny`) — findings stream to stdout for the primary to append to `review-log.md`. The reviewer emits the `## Review Round` header first, then each finding as determined, with the verdict last, so partial stdout is already-useful findings if the reviewer is killed. On timeout, the primary salvages whatever review text was emitted, marks it `PARTIAL`, and applies a heuristic: re-run once at full budget if the partial carries ≥1 finding or was killed >120s in; otherwise escalate.
**Rationale:** The reviewer is a quality gate and must be thorough — the original failure was starvation (a 300s cap on pro, which needs time to read/re-grep/re-examine large artifact sets), not excess activity. A larger budget with incremental emission means a cutoff yields at least a partial review rather than nothing (the original failure produced zero findings). Keeping the reviewer read-only preserves the separate-model review guarantee; streaming findings to stdout is lighter-weight than granting scoped `edit` permission and avoids uncertain OpenCode path-scoped permissions.
**Alternatives considered:** (a) Throttling the reviewer ("read each file once, don't re-grep") — rejected; fights the goal of thoroughness, and re-examination on new information is legitimate review behavior. (b) Granting the reviewer scoped `edit` permission to write `review-log.md` directly — rejected; weakens the read-only guarantee, and opencode path-scoped permissions are uncertain. (c) Keeping the 300s cap and just warning after — rejected; the failure was genuine starvation, not a quality problem; the cap needed raising.

## Decision: Tier-confirmation gate for non-fast-track agents (harden-instruction-surface, 2026-06-16)

**Decision:** A non-fast-track agent MUST propose its change tier together with a plan and obtain explicit operator confirmation BEFORE beginning execution (delegating apply, editing implementation code, or otherwise mutating project state). Producing the plan is NOT gated — the gate sits between planning and execution, not before planning. An agent WITH an explicit fast-track/autonomy grant remains free to self-classify and proceed per `ai-docs/fast-track-workflow.md`. If the operator is unavailable, the agent does NOT execute — it reports the proposed tier and plan and waits, rather than defaulting to autonomous execution.

**Why now / why this shape:**
- The old AGENTS.md `## Change tiers` text told agents to "classify every change yourself and state the tier" — an agent recently self-classified a change and began applying edits without ever showing a plan. The instruction actively licensed unsafe autonomy.
- The gate between planning and execution (not before planning) preserves the agent's ability to surface its reasoning in the plan — a written proposal/tier recommendation is how the choice is made visible — while closing the execution loophole.
- Fast-track remains self-directed: the gate distinguishes autonomy mode (which is standing, opt-in, and trust-gated) from process-weight scaling (which is also standing). They were independent concerns; bundling them under a single "classify yourself" instruction was the original defect.
- Operator-unavailable is explicitly fail-safe: wait, don't default to autonomous.
- The requirement is written into a new `openspec/specs/tier-confirmation-gate/spec.md` capability spec and enforced by the AGENTS.md text itself (the instruction IS the mechanism).

**Motivation:** A 2026-06-16 audit of the first-load instruction surface discovered that the `## Change tiers` guidance was the single load-bearing unsafe instruction — it authorized any agent (regardless of fast-track status) to self-classify and begin executing with no operator visibility. The fix closes a real demonstrated hazard (agent self-classified and began applying edits) via a lightweight instruction-text change that surfaces on every agent's first load. Archive: `openspec/changes/archive/2026-06-16-harden-instruction-surface`. Modified spec: `openspec/specs/apply-convergence-guard/spec.md` (recovery routing now explicitly holds in all autonomy modes). New spec: `openspec/specs/tier-confirmation-gate/spec.md`. Extended spec: `openspec/specs/commit-test-gate/spec.md` (verify re-run single-sourced from `scripts/test-cmd`; AGENTS.md acknowledges the shipped hook).

**Alternatives considered:** The old "classify every change yourself and state the tier" self-authorization — rejected as the cause of the observed safety incident. Requiring confirmation even for fast-track — rejected; fast-track is explicitly opt-in and trust-gated by the operator; over-gating it would defeat its purpose.

## Decision: Delegated `opencode run` closes stdin (`< /dev/null`) (2026-06-16)

**Decision:** Every delegated `opencode run` invocation in the OpenSpec workflow — the propose reviewer, the apply executor, the archive executor, and the verify fix-executor — closes stdin (`< /dev/null`). With no TTY attached, opencode's permission system defaults `external_directory` and `doom_loop` to `ask`; an unanswerable `ask` blocks on stdin indefinitely. Closing stdin makes the prompt auto-reject (fail-fast) — the fail-*safe* direction — and neutralizes every prompt source generically, not just the one confirmed by probes. Each invocation also passes `--dir <repoRoot>` for path-resolution hygiene (does not itself fix the hang).

**Why now / why this shape:**
- Empirically proven: live probes against opencode 1.17.7 confirmed every stdin-open `external_directory` prompt hung (killed at 40s timeout); every stdin-closed prompt auto-rejected in ≤22s. This is the *only* single reliable lever.
- Inline file-contents workaround rejected: does not scale to large changes; the agent still has tools and can still trigger a prompt.
- `--dangerously-skip-permissions` rejected: accepts the auto-approve fail-open for read-only reviewer but is unsafe for write-capable executors (would auto-approve out-of-tree writes), and defeats scoped per-agent containment.
- Per-agent config alone (no stdin close) rejected: probes hung even with `external_directory` configured in agent frontmatter — config-alone was insufficient in practice. The stdin close is generic (neutralizes whatever prompts) whereas per-agent config must be pinned per vector.

**Motivation:** A real reviewer session hung ~30 minutes on an `external_directory` permission prompt with no TTY attached. Delegated work must not silently hang. Archive: `openspec/changes/archive/2026-06-16-harden-delegation-robustness`. New spec: `openspec/specs/noninteractive-delegation-safety/spec.md`.

## Decision: Per-agent `external_directory` posture, split by destructive capability (2026-06-16)

**Decision:** So the stdin-close auto-deny does not starve legitimate access, each delegated agent carries explicit `external_directory` permission in its frontmatter, split by whether the agent has destructive capabilities: the read-only reviewer (`bash: deny`, `edit: deny`) gets `external_directory: allow` (zero added risk — it still cannot write/execute); the write-capable executors (apply and archive, both `bash: allow`, `edit: allow`) get `external_directory: { "*": deny, "/tmp/**": allow }` — the apply-executor's `/tmp` convergence-helper write keeps working while every other out-of-tree path is hard-denied (containment win against a stray out-of-tree `rm`). The pattern order is load-bearing: opencode is last-match-wins, so the catch-all `"*"` must be listed *first* (probe Q3 demonstrated the inverse order silently denies `/tmp`). The archive executor shares the apply executor's scoped-deny config for uniformity, not because it needs the `/tmp` carve-out itself.

**Why now / why this shape:**
- Blanket `external_directory: allow` for all three rejected: removes out-of-tree containment for write-capable executors (the operator's `rm -rf /` concern).
- Global `~/.config/opencode/opencode.json` config rejected: untracked harness state, violates the prime directive (all project state in tracked agent-neutral files) and is not propagatable. Per-agent markdown frontmatter is tracked and is exactly what the single-source "change 2" carries.
- `question: deny` deferred: its runtime behavior was not probed; the stdin close covers it generically.

**Motivation:** The stdin-close auto-deny makes every unanswerable prompt fail. Without explicit allows, even legitimate out-of-tree reads (reviewer examining an artifact) or `/tmp` writes (apply-executor convergence state) would silently fail. The permission split preserves needed access while tightening containment for write-capable executors. Archive: `openspec/changes/archive/2026-06-16-harden-delegation-robustness`. Spec: `openspec/specs/noninteractive-delegation-safety/spec.md` (Requirement: Delegated agents leave no reachable ask-permission).

## Decision: The non-convergence canary is an impl-module + instruction-frozen test (2026-06-16)

**Decision:** The converge-guard canary was rebuilt from a single self-referential file (where `tasks.md` told the executor to edit the same file holding `assert 1 + 1 == 3`, so an honest executor turned it green and never exercised the STOP path) into two files: an editable implementation module (`canary_impl.py`) and a frozen, instruction-gated test (`test_canary.py`) that imports the impl, calls it ONCE capturing a single return value, and asserts that value against contradictory predicates (e.g. `result = add(1, 1); assert result == 2 and result == 3`). Capturing once is load-bearing: a single captured int cannot equal both values, even if the impl is stateful. `tasks.md` lists only the impl as editable and marks the test do-not-edit. The freeze is instruction-gated (hardens the *honest* edit-the-assertion shortcut, not a malicious rewrite), which is sufficient for the canary's purpose. Expected outcome: a declared `### NON-CONVERGENCE BLOCKER` (trigger a, b, or c — whichever the guard reaches first), not a green or a timeout.

**Why now / why this shape:**
- Filesystem-enforced freeze (opencode per-file `edit` deny or a plugin) rejected: overkill for a fixture; instruction-gated freeze is sufficient for the canary's purpose (hardening the honest shortcut, not a malicious rewrite).
- The canary cannot promise a single fixed trigger signature: pytest assertion-rewriting re-renders failing values as the impl changes, so the trigger may be a, b, or c. The guarantee is a declared stop, not a specific trigger.
- Single self-referential file rejected: it was the original defect — the executor was told to edit the file holding the impossible assertion.

**Motivation:** The prior canary was gameable — its own `tasks.md` directed the executor to edit the file containing the impossible assertion, so an honest executor following its instructions could turn the test green by editing the assertion itself, never exercising the convergence guard's STOP path. A canary that doesn't actually trigger non-convergence is a dead test. Archive: `openspec/changes/archive/2026-06-16-harden-delegation-robustness`. Extended spec: `openspec/specs/apply-convergence-guard/spec.md` (ADDED Requirements: hardened, instruction-gated canary).

## Decision: Independent multi-model verification passes as hard gates in the verify step (2026-06-16)

**Decision:** The verify step now runs independent multi-model verification passes as hard gates, layered after the orchestrator's own MANDATORY behavioral self-review and before the artifact/spec mapping checklist and report. A new delegated `openspec-verifier` agent (read-only on files, bash-capable, `edit` denied) runs the same behavioral review the self-review runs and emits a machine-discriminable verdict. The pass chain depends on the orchestrator's platform: a Claude Code orchestrator runs self → `deepseek-v4-pro` → `deepseek-v4-flash`; an OpenCode orchestrator (already deepseek-pro) runs self → flash only. The self-review is unchanged and still not delegated.

**Why now / why this shape:**
- **The passes are additive, not a replacement (D1):** The orchestrator's self-review ("do not delegate this") is preserved as pass 1; the delegated passes are additional independent confirmations. Rejected: replacing the self-review with delegated passes (loses the orchestrator's from-disk judgment; contradicts the operator's sequencing). Rejected: running delegated passes after the artifact checklist (would run the checklist before the behavioral hard gate).
- **Platform-specific chain by model diversity (D2):** A Claude orchestrator (Anthropic model) gains maximum diversity from both deepseek tiers; an OpenCode orchestrator already runs deepseek-pro, so a second pro pass is near-duplicate — only the cheaper flash tier adds an independent view. Rejected: identical chain on both platforms (wastes a redundant pro pass under OpenCode).
- **One agent, model chosen per invocation (D3):** A single `.opencode/agents/openspec-verifier.md` with frontmatter default `model: deepseek/deepseek-v4-flash`. The Claude path invokes it via `opencode run --agent openspec-verifier --model deepseek/deepseek-v4-pro` then `--model deepseek/deepseek-v4-flash`; `--model` overrides the frontmatter. Rejected: two agent files (`-pro` / `-flash`) — duplicated body prompt that must be kept in sync.
- **Third permission category (D4):** The verifier (`bash: allow`, `edit: deny`) takes executor-style `external_directory` containment (`"*": deny`, `/tmp/**: allow`), not the read-only reviewer's `external_directory: allow`, because `bash: allow` makes it write-capable in practice. It is strictly less capable than the existing apply-executor (no `edit`, no `task`), so it introduces no new trust boundary. Rejected: reusing the `openspec-reviewer` with `bash` flipped to allow (conflates the read-only propose auditor with a test-running verifier).
- **Machine-discriminable verdict, judged from disk (D5):** The verifier emits a `## Verify Pass — <model>` / `VERDICT: READY` or `VERDICT: NEEDS REVISION` / `### Defects` block. The orchestrator asserts the real verifier ran (grep stderr for `Falling back to default agent` → escalate; confirm `## Verify Pass` + `VERDICT:`), then judges every finding from disk — findings are leads to confirm, not gospel. A demonstrably false finding may be overruled only with recorded rationale.
- **Rerun-failed-and-after recovery (D6):** On NEEDS REVISION (confirmed from disk), fix via the existing re-delegation path, then rerun the failed pass and every pass after it — never the earlier ones. Accepted trade-off: a fix for a late pass is not re-checked by earlier passes; the orchestrator still judges every later pass from disk, and flash (cheapest) is always last. Three-cycle bound per pass → operator escalation.
- **noninteractive-delegation-safety extended (D7):** The Claude Code verifier `opencode run` invocations join the enumerated set that SHALL close stdin; the OpenCode Task-tool spawn is exempt. The third permission category is stated explicitly in the noninteractive-delegation-safety spec's agent-permission taxonomy.
- **Audit trail (D8):** The verification report and `notes.md` record, per pass, the model, verdict, and any defect caught (attributed to the pass that surfaced it), making the multi-model gate auditable.

**Motivation:** Verify is the final gate before a change is declared "good to release," yet it rested on a single reviewer — the orchestrator's own behavioral self-review. One model's blind spots therefore decided release. Independent models reliably catch different falsifiable defects — a discipline this project already follows, and one that has repeatedly caught real blocking defects the primary missed. The highest-stakes gate should not be the one place that relies on a single set of eyes. Archive: `openspec/changes/archive/2026-06-16-verify-multimodel-gate`. New spec: `openspec/specs/verify-multimodel-gate/spec.md`. Modified spec: `openspec/specs/noninteractive-delegation-safety/spec.md`.

## Decision: Commit-test-gate hook wiring live-verified — W0 smoke passed (2026-06-16)

The previously BLOCKING open-question — does the hook fire on `git commit`, does exit 2 block the commit, does `$CLAUDE_PROJECT_DIR` expand — is RESOLVED. Reliance on the commit-test-gate, and on the same hook-wiring mechanism W1's scaffold guard (`scaffold_check.py`) will use, is unblocked.

## fix-sync-mechanism (2026-06-17)

**Decision:** The scaffold→downstream sync mechanism was built with four new files: `scripts/sync_scaffold.py` (copies manifest-listed files byte-identical, with AGENTS.md span-replace preserving per-repo title/project-context/tail), `scripts/scaffold_manifest.txt` (self-listed authoritative inventory), `scripts/scaffold_check.py` (PreToolUse guard, exit 2), and `scripts/test_sync_scaffold.py` (unit tests). Two blocking bugs from the prior frozen `scaffold-sync` plan are fixed — the guard now exits 2 (not 1) and the DO NOT EDIT header subsystem is cut entirely — making the mechanism actually functional and idempotent on repeated sync.

**Why now / why this shape:**
- **Header subsystem cut (D4):** The prior plan injected a format-specific "DO NOT EDIT" header into every synced file and stripped it back out before diffing in `--check`. This subsystem is removed entirely — the sync script injects nothing, `--check` for regular files is a plain byte compare, and the single human-facing "scaffold-managed — edit upstream" note lives once in AGENTS.md and as ordinary manifest content. Even one uniform header would reintroduce an insert/strip round-trip to keep idempotent, for an advisory string whose real enforcement is the hook. Editors are overwhelmingly agents, gated by the commit guard. One-uniform-header alternative was considered and rejected for this reason.
- **Guard exits 2, not 1 (D5/B1):** `scaffold_check.py` exits 2 when a scaffold-managed file is staged for commit — the only Claude Code PreToolUse exit code that blocks the tool. The prior plan's `exit(1)` was a silent no-op. The exit-2 blocking convention was live-verified in W0 via the commit-test-gate hook. The guard resolves the manifest relative to its own file, not cwd, so it requires no scaffold path and makes no cwd assumption. Coverage is limited to Claude Bash-tool commits (M1) — documented, not engineered around; operator-terminal and opencode/deepseek executor commits bypass it, and `--no-verify` is the sanctioned escape.
- **Manifest: plaintext, self-listed (D2):** `scripts/scaffold_manifest.txt` — one repo-relative path per line, `#` comments, lists itself and `scaffold_check.py` (downstream hooks depend on both). Excludes per-repo/volatile state. Python (not shell) for the sync script because AGENTS.md span-replace needs multi-line regex.

**Motivation:** Four prior scaffold changes sat inert in downstream repos (extrends, psc-monitor) with no propagation mechanism. A prior `scaffold-sync` plan existed but a principal review found two runtime bugs — the non-blocking guard exit code and non-idempotent AGENTS.md header — that would corrupt files on repeated use. This change builds the mechanism correctly and minimally so the forthcoming one-time propagation (W6) has a tool it can trust. Archive: `openspec/changes/archive/2026-06-17-fix-sync-mechanism`. New spec: `openspec/specs/scaffold-sync-mechanism/spec.md`.

## lifecycle-gates (2026-06-17)

**Decision:** The verify gate was tier-scaled, two new quality/security gates were added to the verify step, the archive-executor bodies were hardened for RENAMED spec requirements, and a new executor-pair body-agreement guard was introduced. Six audit findings (E1, B2, D-ii, E3, D-i, C4) were folded into one MEDIUM change. The multi-model verification passes now apply to MEDIUM/COMPLEX changes only — SMALL changes do their own verification per AGENTS.md (optionally one flash pass) and do not invoke the verify skill. A simplicity/quality gate (harness-neutral, after verifier passes, before artifact checklist) and a conditional security gate (for auth/credentials/data/external-API changes) were added. Both archive-executor bodies now handle renames (RENAMED FROM:/TO:). A `scripts/test_executor_body_agreement.py` guard byte-compares each `.claude`/`.opencode` executor-pair body and is manifest-listed for W6 propagation.

**Why now / why this shape:**
- **Tier-scale, not remove (B2/D-ii):** The multi-model passes were previously unconditional — a one-line SMALL change triggered the full self→pro→flash chain (two 780s invocations). The fix exempts SMALL and preserves the full chain for MEDIUM/COMPLEX. The verify skill self-documents its MEDIUM/COMPLEX-only applicability and carries NO SMALL-branch logic — a SMALL-branch inside the skill would be dead code since SMALL never reaches the file. AGENTS.md is the single home for SMALL verify behavior. Rejected: adding SMALL-branch logic inside the verify skill (dead-code contradiction flagged by the reviewer); removing passes entirely (defeats the original gate's purpose for non-trivial changes).
- **Simplicity gate is harness-neutral (E1):** Under Claude Code, the orchestrator invokes the `simplify` (or `/code-review`) skill on the diff. Under OpenCode (where no such skill exists), the orchestrator uses a concrete 4-item checklist — (a) code duplicating existing functionality, (b) single-use abstractions, (c) dead/unreachable code, (d) over-parameterization beyond scope. The checklist IS the OpenCode instruction — not "perform an equivalent review" (which gives the executor nothing concrete). Findings route through the existing defect re-delegation path. The gate does not block on pure style nits. Rejected: naming only the Claude skill without an OpenCode path (violates harness-neutrality); blocking on style (contradicts the verify skill's existing "don't nitpick style" rule).
- **Security gate is conditional and tiered (E3):** Only triggers for changes touching auth, credentials/secrets, persisted data, or an external API/network surface. Hard gate at COMPLEX, recommended at MEDIUM. Harness-neutral same pattern: Claude invokes `security-review`, OpenCode uses a concrete 5-item checklist (authn/authz bypass, credential leakage, injection, unsafe deserialization, missing destructive-operation guards). Rejected: making it unconditional (wastes a review on instruction-only changes); making it hard-gate at MEDIUM (over-gates medium-risk instruction changes).
- **RENAMED in both archive-executor bodies (D-i):** The archive skill and `openspec-sync-specs` both handle renames, but both executor agent bodies said only "additions, modifications, and removals" — a RENAMED requirement synced via archive could be silently dropped. The fix adds "and renames (RENAMED, FROM:/TO: format — see `openspec-sync-specs`)" to both `.claude/agents/archive-executor.md` and `.opencode/agents/archive-executor.md`, with byte-identical wording so the C4 guard stays green.
- **C4 body-agreement guard (`.claude`/`.opencode` executor pairs):** Each executor (apply, archive) exists as a `.claude/` + `.opencode/` pair whose prose bodies are hand-synced with no mechanical check. `scripts/test_executor_body_agreement.py` strips YAML frontmatter, normalizes the one sanctioned divergence (the intro sentence's "the Claude Code counterpart of the OpenCode `@…`" clause), and asserts the remainder is byte-identical. It FAILS when the two bodies drift on any non-intro line. On first run it caught pre-existing drift in both pairs — the apply-executor's `.opencode` body had bullet ordering differences, and the archive-executor's `.claude` body was missing `**completion report**` bold present in `.opencode`. Both were normalized (no semantic change). The test is listed in `scripts/scaffold_manifest.txt` for W6 propagation. Rejected: relying on manual sync indefinitely (the very drift this guard caught proves manual sync fails); a more invasive single-source mechanism (overkill for two near-identical files whose only divergence is a sanctioned intro clause).

**Motivation:** A 2026-06-16 workflow audit surfaced six gaps in the lifecycle gates: the verify multi-model passes were not tier-scaled (wasting two 780s invocations on one-line SMALL changes), no simplicity/quality gate existed anywhere (over-engineering/dead-code flowed through ungated), no security review existed for sensitive-surface changes (auth/data/API changes had no security checkpoint), the archive-executor bodies silently dropped RENAMED spec requirements, and executor-pair bodies were drifting with no guard. The change closes all six in one MEDIUM change. Archive: `openspec/changes/archive/2026-06-17-lifecycle-gates`. Modified spec: `openspec/specs/verify-multimodel-gate/spec.md`.
