# Copier Tool Spike — Research Findings

Date: 2026-06-18  
Tool: copier (Python, copier.readthedocs.io)  
Question: Can `copier update` cleanly express "replace 2 named spans + preserve title/project-context/tail" intra-file?

---

## Q1: How does `copier update` work?

**Algorithm (from docs):**
`copier update` is NOT a simple "apply template and overwrite." It runs a smart diff/patch merge:

1. Clone template at the **old commit** → regenerate a fresh project in `/tmp/project_regen`
2. Diff `project_regen` vs. **actual current project** → captures all local edits the user made
3. Regenerate the project from the **new (latest) template** and apply it in-place
4. Re-apply the captured local-edit diff on top of the newly templated result
5. If the two sets of changes conflict → emit conflict markers (inline `<<<<`/`====`/`>>>>`) or `.rej` files

**Underlying mechanism:** diff/patch (not `git merge`). The project repo is not used as a git merge base; copier regenerates a synthetic "old template" baseline and diffs from that.

**Conflict handling:** `--conflict inline` (default) writes git-style conflict markers in-file. `--conflict rej` writes `.rej` files alongside. Both require manual resolution.

**Source:** https://copier.readthedocs.io/en/stable/updating/ and https://github.com/copier-org/copier/blob/master/docs/updating.md

## Q2: Partial-file / region preservation

**Short answer: No. Copier has no intra-file region concept. It operates at whole-file granularity.**

### What exists at file level

| Feature | What it does |
|---|---|
| `_skip_if_exists` | Skip rendering/overwriting a **whole file** if it already exists in the destination. During `update`, if the file was previously missing it gets created; if present it is left alone. NOT a region mechanism — it skips the *entire* file. |
| `_exclude` | Exclude files/dirs from the template entirely; they are never rendered or updated. |
| `{% if condition %}.filename{% endif %}.jinja` | Conditionally generate a whole file based on answers. Not region-based. |

### The 3-way merge is the only intra-file "preservation" mechanism

When `copier update` runs, local edits to *any span* of a file survive only if:
- The new template version did NOT touch overlapping lines (no conflict → the patch applies cleanly)
- OR the user manually resolves a conflict marker

There is no way to tell copier "always keep lines X–Y verbatim." The diff/patch machinery treats every line equally.

### Jinja variables as a workaround (partial, awkward)

You *can* encode per-repo variable content as copier question answers stored in `.copier-answers.yml`:

```yaml
# copier.yml
project_title:
    type: str
    help: "Repo title"

project_context:
    type: str
    multiline: true
    help: "Per-repo project context section"

repo_appendix:
    type: str
    multiline: true
    help: "Long tail appendix for this repo"
```

Then the template file renders these:

```jinja
# {{ project_title }}

{{ project_context }}

<!-- BEGIN MANAGED SECTION: agents -->
... template-controlled content ...
<!-- END MANAGED SECTION: agents -->

{{ repo_appendix }}
```

On `copier update --defaults`, the stored answers are reused verbatim, so `project_title`, `project_context`, and `repo_appendix` stay unchanged while the "managed section" between the markers gets the updated template content.

**But this requires:**
- Encoding every per-repo prose blob as a copier question answer upfront (at `copier copy` time)
- The `.copier-answers.yml` file must be regenerated by the template (using `{{ _copier_conf.answers_file }}.jinja`) — you cannot manually edit it without breaking copier's diff algorithm
- A long multi-hundred-line appendix tail would need to be a multiline YAML string in answers — ugly

**Sources:** https://github.com/copier-org/copier/blob/master/docs/configuring.md (skip_if_exists, exclude, Jinja variables), https://github.com/copier-org/copier/blob/master/docs/creating.md

## Q3: Carrying code and dotfiles

**Yes, fully.** From the creating docs:

> "The content of the files inside the project template is copied to the destination without changes, **unless they end with `.jinja`**."

This means:
- `.py` scripts → copied verbatim (or templated if `.py.jinja`)
- `.claude/`, `.opencode/` dotfile dirs → copied verbatim, directory structure preserved
- Skill/agent markdown files → copied verbatim (unless suffixed `.jinja`)
- Binary files → copied without modification

**No known limitations** on file type, extension, or hidden directory names. The template can contain any file tree.

**Source:** https://github.com/copier-org/copier/blob/master/docs/creating.md

## Q4: Multi-repo mechanics

**Per-repo registry:** Each generated project gets a `.copier-answers.yml` file (created by the template via `{{ _copier_conf.answers_file }}.jinja`). This records:
- `_src_path`: template URL or local path
- `_commit`: Git tag of the template version used
- All question answers

**Updating N repos:** No built-in batch command. You script it:
```bash
for repo in repos/*/; do
    (cd "$repo" && copier update --defaults)
done
```

**Version pinning:** Each repo's `.copier-answers.yml` pins `_commit` to the template tag used at last update. Running `copier update` advances to the latest template tag by default; `--vcs-ref=<tag>` pins to a specific version.

**Staleness checking:** `copier check-update --output-format json` returns `{"update_available": true/false, "current_version": "...", "latest_version": "..."}` — scriptable for CI automation.

**Source:** https://copier.readthedocs.io/en/stable/updating/, https://copier.readthedocs.io/en/stable/configuring/

## Verdict: Can copier express "replace 2 named spans + preserve title/project-context/tail"?

**Not natively.** Copier has no intra-file region primitive.

**Mapping to your AGENTS.md use case:**

| What you need | Copier can? | Notes |
|---|---|---|
| Replace 2 named managed sections in AGENTS.md on update | Via 3-way patch only | Works if local edits don't overlap the managed sections; degrades to conflict markers if they do |
| Preserve title verbatim across updates | Only if stored as a copier question answer | Requires encoding at `copier copy` time |
| Preserve "Project context" section verbatim | Only as a copier question answer | Same constraint — multiline YAML in answers file |
| Preserve long per-repo appendix tail | Only as a copier question answer | A 200-line appendix becomes a multiline answer blob in `.copier-answers.yml` |
| Guarantee zero conflicts on the non-managed spans | No guarantee | If template changes lines near a preserved span, patch may conflict |

**The honest model mismatch:** Your bespoke script applies explicit named-region semantics ("this span is TEMPLATE-owned, that span is REPO-owned"). Copier's update model is "diff from synthetic baseline, patch forward, resolve conflicts." They're equivalent only when local edits and template edits never touch neighboring lines — which is not guaranteed for a living AGENTS.md file.
