## Task 2 — Bounding options

### Option A: Active/archive split

**Description:** Move resolved decisions + answered questions to archive; keep live files small.  

**Grounded in:** ADR lifecycle (Accepted → Superseded → Archived in Michael Nygard's ADR pattern); changelog "Unreleased vs released" split (Keep a Changelog). The analogy is clear: SUPERSEDED decisions in decisions.md are like "Released" changelog sections — they've served their purpose and belong in an append-only archive log, not the active workspace.

**Concrete mechanics per file:**
- `STATUS.md`: The existing cap rule (3 entries) + a new word-budget constraint (≤150 words per change entry) enforced at archive time. Full narratives go to `ai-docs/archive/status-log.md`. No new mechanism — the archive-executor already handles this; the cap constraint just needs tightening.
- `open-questions.md`: At archive time, shipped sections move to parked-follow-ons.md unless they contain an active BLOCKING item. If the section has a BLOCKING item, it stays; its non-blocking bullets park. Section is condensed to: header + BLOCKING bullet(s) + one-line pointer to `parked-follow-ons.md § <area>`.
- `decisions.md`: Move 2 SUPERSEDED stubs to `ai-docs/archive/retired-notes.md` (they're already nearly empty). For large change-record entries (>100 lines), replace with: `**Summary:** <50 words>. Full rationale: openspec/changes/archive/<change>/notes.md.` The complete entry can move to a `ai-docs/archive/decisions-archive.md`.

**Interaction with structured vs prose asymmetry:**  
- psc-monitor: Already structured — `**Status:** SUPERSEDED` field would enable automated skip-at-boot (or archive-executor check). Easy to apply.
- extrends: Entries lack a `**Status:**` or `**Date:**` field on older sections. The archive-executor cannot distinguish "old but still architecturally durable" from "change-record that can be summarized" without reading the entry in full. Option A is HARDER for extrends without some light structuring first (see Option C below).

**Trade-offs:**
- Pro: Directly reduces the file sizes. Consistent with how CHANGELOG.md and ADR repositories are maintained at scale. No compiler needed.
- Con: Retroactive cleanup of 56 extrends decisions.md entries is ~1–2 hours of work. Risk of losing future recall: if an agent needs to understand why Backfills commit separately from aggregation, it must now navigate to the change archive. The archive file (decisions-archive.md) can itself grow large over time.

### Option B: Headers/index-at-boot + drill-down on demand

**Description:** Boot routing reads only compact index/headers; agent opens full entries when relevant.

**Grounded in:** Progressive disclosure as documented in Claude Code's own docs ("read headers first, then drill into relevant sections"); AGENTS.md already applies this to `decisions.md`. The pattern is well-established in the scaffold itself.

**Concrete mechanics:**

decisions.md: Already implemented. Cost at boot = ~2k tokens (headers). No change needed here.

STATUS.md (proposed extension): Change the AGENTS.md routing from "read in full" to: "read the `## Immediate next action` section and the `## Current state` summary (first bulleted paragraph or table) at boot; read the full `## Latest change` / `## Prior change` sections only when resuming an in-progress change or verifying a specific recent decision." This would reduce STATUS.md boot cost from 7k to ~1–2k tokens for psc-monitor and from 7k to ~1.5k tokens for extrends, even without changing the file content.

open-questions.md (proposed extension): Change routing from "read in full" to "read all section headers at boot; read the full section only when working in that area." But this directly undermines the purpose of open-questions.md — blockers must be visible at boot, not behind a header-scan. A partial version: read in full only the `## Needs user action` and any section containing `**BLOCKING**` marker at the section-header level — but this requires the section header to flag itself (e.g., `## Per-lane scoring — BLOCKING`), which is a light structuring change.

**Interaction with structured vs prose asymmetry:**  
- psc-monitor: The format is already structured enough for programmatic BLOCKING-detection. Boot routing could skip sections whose headers don't contain "BLOCKING".  
- extrends: Section headers don't reliably signal BLOCKING status (e.g., "Stage activations + automated board labeling (shipped 2026-06-16)" contains 2 BLOCKING items but the header says nothing). Agents reading only the header would miss these blockers. Requires either annotating headers or reading all sections in full — which returns to the current state.

**Trade-offs:**
- Pro for STATUS.md: Zero content changes needed. Immediate boot-cost reduction by changing the AGENTS.md routing instruction.
- Pro for decisions.md: Already done — no action needed.
- Con for open-questions.md: The whole point of open-questions.md is that blockers are visible at boot. Headers-only routing breaks this invariant for extrends's prose format. Would require header-level BLOCKING annotations to work safely.
- Ground: The progressive-disclosure pattern is proven for stable/large reference files (decisions.md). It is a poor fit for the active-blockers file (open-questions.md) where missing a blocker is a correctness failure.

### Option C: Status frontmatter / structured fields

**Description:** Add `**Date:** YYYY-MM-DD` and `**Status:** ACTIVE / SUPERSEDED / STABLE` to extrends decisions.md entries, aligning it with psc-monitor's format.

**Grounded in:** ISO 25010 documentation quality standard ("a record should be self-describing"); ADR repositories conventionally use status fields (Proposed / Accepted / Deprecated / Superseded — e.g., adr.github.io). psc-monitor already follows this pattern for its own entries.

**Concrete mechanics:**
- Going forward: all new extrends decisions.md entries must include `**Date:**` and `**Status:** ACTIVE`.
- Going-forward only (no retroactive backfill required): `**Status:** SUPERSEDED` added to the 2 existing stubs (2-line edit); `**Status:** STABLE` added to the ~15 foundational pre-dated entries (identifies them as "not a regression risk, don't re-read on every change").
- Enables the archive-executor to cheaply identify archivable entries: "any entry with `**Status:** SUPERSEDED` or older than 90 days with `**Status:** STABLE` and no cross-references in recent changes can be archived."

**Interaction with structured vs prose asymmetry:**  
- This RESOLVES the asymmetry going forward. After applying Option C, both repos follow the same convention.
- retroactive backfill of 56 sections is optional but not required for the rule to provide value on new entries.

**Trade-offs:**
- Pro: Low effort (2-line changes for existing SUPERSEDED stubs, 1-line per new entry). Enables future archiving without a compiler. Standardizes both repos.
- Con: Doesn't reduce current file size — it's enabling infrastructure for Option A. Adds a tiny per-entry overhead (~10 characters). Retroactive backfill is non-trivial for 56 entries.

### Option D: Caps / rotation policy

**Description:** Keep last N entries / archive older than X days; hard per-entry word budget.

**Grounded in:** Log rotation conventions (logrotate, newsyslog); CHANGELOG.md "Keep the unreleased section ≤ N entries" patterns used in npm/ruby ecosystem. The pattern: define a maximum size budget, enforce at write time (archive-executor), audit at read time.

**Concrete mechanics:**
1. **STATUS.md cap + word budget**: Existing rule = 3 entries. Enhancement: each entry ≤150 words. Archive-executor enforces both at archive time.
2. **open-questions.md section budget**: Each section ≤600 words total. Excess bullets must be labeled BLOCKING or they move to parked-follow-ons.md.
3. **decisions.md entry budget**: Each entry ≤300 words. Entries exceeding this are summarized in place (≤300 words + pointer to change archive).

**Interaction with structured vs prose asymmetry:**  
- The word budget applies identically to both formats — it's format-agnostic.
- However, enforcing the decisions.md entry budget on extrends requires the archive-executor to read each entry in full (to assess whether it exceeds budget). For psc-monitor's structured format, the `**Rationale:**` field length is a simpler proxy.

**Trade-offs:**
- Pro: Simple, auditable, format-agnostic. Directly bounds file growth.
- Con: Arbitrary word budgets can create perverse incentives (agents writing terse-but-incomplete rationale to fit the budget). The archive-executor needs to compress entries intelligently, not just truncate.

## Task 3 — Recommendation

### Recommended combination: A + D + C(forward-only)

**The single most effective action is tightening the enforcement of two rules that already exist in AGENTS.md but are being under-enforced by the archive-executor:**

**Change 1 (STATUS.md): Add a per-entry word budget to the existing cap rule.**

Current rule: "at most 3 most recent ## Latest change / ## Prior change paragraphs; older entries move to ai-docs/archive/status-log.md."

Tightened rule: add "Each change-entry section is a 1–3 sentence (≤100–150 word) headline summary: what shipped, key verify outcome, where to find details. The full change narrative is in the change archive (openspec/changes/archive/). Prose beyond this budget moves to status-log.md at archive time."

This alone would reduce extrends STATUS.md from ~7k tokens to ~2k tokens. The "Latest change" paragraph for add-correctness-signals is ~500 words; a 100-word summary exists in the first 2 sentences. psc-monitor's "Latest change" entry is already well within 200 words — the rule is redundant there but adds no harm.

**Change 2 (open-questions.md): Tighten the horizon-split rule's archive-time behavior.**

Current rule: "Resolved items move to ai-docs/archive/retired-notes.md. Deferred/monitored/low-priority items move to parked-follow-ons.md."

Tightened rule: add "At archive time, a shipped-change section in open-questions.md is reduced to: (a) any BLOCKING items verbatim, (b) a one-line pointer for each non-blocking item that parks: `Tune-after-evidence items → parked-follow-ons.md § <area>`. A section with no remaining BLOCKING items is replaced entirely by the pointer line and pruned to < 3 lines. The archive-executor is NOT permitted to leave non-blocking bullets from shipped sections in open-questions.md."

This would reduce extrends open-questions.md from ~7k tokens to ~3k tokens (the 7 BLOCKING items + necessary context + pointers). It does NOT reduce visibility of real blockers — those stay in full.

**Change 3 (decisions.md): Light structuring going forward (no retroactive work required).**

Add to AGENTS.md: "All new decisions.md entries in BOTH repos must include `**Date:** YYYY-MM-DD` and `**Status:** ACTIVE`. When adding a new entry that is a change-record (fix-*, add-*, tune-*), the entry is bounded to ≤300 words; refer to `openspec/changes/archive/<name>/` for the full rationale."

For the 2 existing SUPERSEDED stubs in extrends, add `**Status:** SUPERSEDED` at the next archive (2-line change). No retroactive backfill of all 56 entries required.

### What changes in the boot routing (AGENTS.md)

The scaffold AGENTS.md (which propagates to both repos) needs three sentence-level additions to the existing rules:

1. In the STATUS.md cap rule paragraph: append "Each change-entry section is ≤150 words; details stay in the change archive."
2. In the open-questions.md horizon-split rule paragraph: append "At archive, shipped-section non-BLOCKING bullets immediately park to parked-follow-ons.md; the archive-executor may not leave non-blocking shipped-change bullets in open-questions.md."
3. In the State section: append "New decisions.md entries include **Date:** YYYY-MM-DD and **Status:** ACTIVE; change-record entries (fix-*, add-*, tune-*) are capped at ≤300 words."

These go in the scaffold AGENTS.md and propagate to both downstream repos on the next openspec-sync-specs run.

### Main risk

**Losing live recall of tune-after-real-runs context.** An agent freshly booted on W25 (the first all-stages-on cron run) will no longer see in open-questions.md the 5+ bullets about unvalidated defaults (`wsv_min_independent_corroboration=2`, `entity_prune_after_runs=4`, etc.) because they will have been parked. An agent beginning work in the lane-scoring area without loading `parked-follow-ons.md § lane-scoring` may tune a knob without knowing the tune-after-W25 constraint.

**Mitigation:** The pointer line left in open-questions.md (e.g., "Lane/breadth and correctness knobs: tune-after-W25 items → parked-follow-ons.md § lane-scoring-and-breadth") makes the context one hop away, not invisible. The AGENTS.md instruction to load the relevant parked-follow-ons area when starting work in that area already covers this — the gap is that agents sometimes skip loading parked follow-ons because nothing in open-questions.md signals "you should." The one-liner pointer directly addresses this.

A secondary risk is retroactive application creating gaps: if old verbose STATUS.md entries are truncated by the next archive-executor run, some "why" context that was in the STATUS.md narrative (not in the change archive) may be lost. Mitigation: the archive-executor should check the change archive exists and has a notes.md with the relevant rationale before truncating — if the change archive is thin, copy the salient context there first.
